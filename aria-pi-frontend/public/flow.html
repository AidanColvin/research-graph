How a search becomes a sourced report

Every box is a real stage in the stack. The moving dots trace one packet of information from a typed sector to a downloadable, citation-checked report.

1 · User Browser

Types a sector — anything from Oncology to HVAC to Craft Beer — and presses Generate on the public page.

{ sector: "HVAC" }

2 · Next.js API Proxy /api/run-pipeline

The frontend POSTs the term to its own route. Marked force-dynamic + no-store, so no stale report is ever served from cache — every search is fresh.

POST → cache: "no-store"

3 · Backend Orchestrator FastAPI · Vercel

Resolves the term to companies via canonical_sector():

• curated — known sector → hand-picked list  |  discovered — niche term → live SEC EDGAR full-text search  |  default — fallback anchors

resolution + [ company, company, … ]

4 · Live Source Fetch parallel · no API keys

For each company, the orchestrator pulls primary-source data in parallel:

SEC EDGARXBRL financials, submissions & filings
ClinicalTrials.govtrials & collaborators
PubMed (Entrez)co-authorship & COI disclosures
NIH RePORTERfederal grants
raw facts + source URLs

5 · Report Builder + Verifier deterministic

Assembles Sections 1–7. The source tagger double-checks every claim against ≥2 sources (AMA style) and flags anything unverified. A generated_at timestamp is stamped on.

JSON report { report_meta, sections, references, _meta }

6 · Render Report.tsx

The frontend normalizes the JSON, builds an AMA citation index (numbering every unique URL), and paints the report — superscript citations, tables, profile cards, references.

on-screen report + ¹²³ citations

7 · Download Markdown · PDF · Word

One click flattens the report into a format-neutral block IR, then renders the same content + citation numbers to three files — formatting stays identical across all three.

.md  ·  .pdf  ·  .docx
